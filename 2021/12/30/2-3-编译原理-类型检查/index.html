2.3 编译原理 -- 类型检查

2.3 Type checking（类型检查）

经过 Parsing(词法分析 & 语法分析)阶段后，我们得到了源文件的 AST 树，表明了此时的源码文件已经不存在 词法、语法错误等问题

后续将要执行的类型检查，会遍历 AST 中的节点，对每一个节点的类型进行检验，找出其中存在的 语法 & 类型错误；

同时在这个过程中可能会改写 AST，剔除不会被执行的代码、优化代码、改写 make, new 等关键字

2.3.1 强类型 Strong 与 弱类型 Weak

强类型：
-> 在编译过程中有更严格的类型限制
-> 编译器需要发现 变量赋值、返回值、函数调用时的类型错误
-> 典型的强类型语言：C#, Java, Golang

弱类型：
-> 较为宽松的类型限制
-> 内置隐式的类型转换
-> 在发现类型错误时，可能会在 runtime 中进行隐式的类型转换
-> 隐式的类型转换可能会造成 runtime error
-> 典型的弱类型语言：PHP, JavaScript

需要思考的问题：

  1. 类型的转换是 显式 还是 隐式 ？
  2. 编译器是否会帮助编程人员 推断 变量的类型 ？

2.3.2 静态类型 Static 与 动态类型 Dynamic

静态类型
-> 静态类型检查
-> 基于对源代码的分析来确定运行程序类型安全的过程
-> 减少了程序在运行时的类型检查
-> 在编译期间能够约束变量的类型

动态类型
-> 动态类型检查
-> 在 运行时 确定程序类型安全的过程

静态类型，动态类型并不冲突：

  1. 静态类型编程语言 如：Java，既在编译期间进行了类型检查，又在 runtime 中使用 反射机制，根据对象的类型动态执行方法
  2. 动态类型编程语言 如：JavaScript 通过 Flow 工具，也可以在编译期间发现代码中的类型错误

2.3.3 执行过程 – golang 源码实现

golang 的编译器在编译期间使用 静态类型检查 确保了程序运行的类型安全

golang 较为特殊的一点是：interface{}（可以将其看做简易的 泛型）

当需要将 interface{} 转换成具体类型时，golang 还会在 runtime 中进行 动态类型检查，如果遇到无法发生转换的情况就会发生程序崩溃

Main() 中的类型转换逻辑

src/cmd/compile/internal/gc.Main() 在完成 词法 & 语法分析后，会执行以下操作进行类型检查：

  1. 检查 常量、类型、命名、函数声明的类型，通过 gc.typecheck() 函数
  2. 检查 变量赋值 的类型，通过 gc.typecheck() 函数
  3. 检查 函数体 的类型，通过 gc.typecheckslice() 函数
  4. 检查 hashMap 中的 key-value 类型，通过 gc.checkMapKeys() 函数

make关键字 OMAKE

golang 内置 make() 函数，用于创建 切片、哈希、channel。

类型检查阶段会根据创建的类型，将 make() 替换为特定的函数：

  1. 检查 make 的第一个类型参数
  2. 根据类型不同，进入不同的分支：TSLICE、TMAP、TCHAN

case TSLICE：

  1. 获取参数：len = args[1] 切片长度, cap = args[2] 切片容量
  2. len, cap 校验
  3. 设定节点类型为 OMAKESLICE

case TMAP:

  1. 获取参数：len = args[1]
  2. 设定节点类型为 OMAKETYPE

case TCCHAN:

  1. 获取参数：len = args[1] 缓冲区大小
  2. 设定节点类型为 OMAKECHAN

Conclusion

  1. 类型检查是 golang compile 的第二个阶段，在 词法&语法分析后，我们得到了每个源文件对应的抽象语法树，随后的类型检查会遍历 AST 中的节点，对每个节点的类型进行校验，找出其中存在的 语法错误 导致的 类型错误。

  2. 对 AST 进行改写，如：make() 关键字; Slice、Map、Channel 的处理

  3. make() 和 new() 内置函数，不会直接对应某些函数的实现，而是在编译期间被转换成真正存在的其他函数
